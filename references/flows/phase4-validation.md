# 第四阶段：自动校验与修复

**目标**：确保所有章节完成且字数达标，全程无需用户介入。

---

## 校验流程

### 1. 初始化校验

读取 `03-写作计划.json`，将项目 `status` 更新为 `"validating"`。

### 2. 逐章检查

对每一章执行以下检查：

1. **文件存在性**：检查 `filePath` 指定的文件是否存在
2. **字数检查**：使用脚本检查字数
   ```bash
   python scripts/check_chapter_wordcount.py <章节文件路径>
   ```
3. **爽点检查**：
   - 本章是否有核心爽点？
   - 爽点密度是否达标（每2000字≥1个爽点）？
   - 是否有让读者想截图的金句？
4. **情绪曲线检查**：
   - 本章是否有"压抑→释放"的弧线？
   - 憋屈是否持续过长（超过3章需释放）？
5. **人物一致性检查**：
   - 人物对话语气是否符合其性格设定？
   - 主角能力值、状态是否与上章衔接？
6. **伏笔检查**（非第1章）：
   - 本章规划中需要回收的伏笔是否已回收？
   - 本章埋下的伏笔是否已记录到章节摘要？
7. **更新 JSON**：将 `wordCount`、`wordCountPass`、校验结果写入对应章节记录

### 3. 汇总结果

- **全部通过**（所有章节文件存在且 `wordCountPass == true` 且有爽点）→ 更新项目 `status` 为 `"completed"`，进入步骤5
- **有不通过** → 进入步骤4

### 4. 自动修复（最多3轮）

对每个不通过的章节：

1. 将 `status` 设为 `"failed"`，`retryCount` 加 1
2. 回到第三阶段对该章节执行逐章创作流程（写前分析→撰写→润色→收尾）
3. 重写完成后重新检查字数和爽点
4. 更新 `03-写作计划.json`

**校验维度（完整版）**：

| 维度 | 目标 | 不合格处理 |
|-----|------|----------|
| **字数** | 2000-3000字/章 | 扩充或压缩 |
| **爽点密度** | 每2000字≥1个爽点 | 增强爽感描写 |
| **金句** | 每章至少1个金句 | 强化结尾/高潮 |
| **开头钩子** | 前20%有即时爽点 | 重写开头 |
| **情绪曲线** | 有压抑→释放弧线 | 调整节奏 |
| **人物语气** | 对话符合性格设定 | 调整对话 |
| **伏笔回收** | 规划回收的伏笔需兑现 | 补充伏笔回收 |

循环规则：
- 最多执行 **3轮** 校验-修复循环
- 超过3次仍不合格的章节，保留记录但不阻塞流程
- 所有修复完成后更新项目 `status` 为 `"completed"`

### 5. 完成报告

向用户展示创作完成总结：

```
📊 《[小说名称]》创作完成

总章数：[X] 章
总字数：[X] 字
完成率：[X]%

各章节状态：
✅ 第1章：[标题]（[字数]字）- [名场面类型]
✅ 第2章：[标题]（[字数]字）- [名场面类型]
...

项目文件夹：./web-novels/[timestamp]-[小说名称]/

核心爽点：[核心爽点]
金手指：[金手指]
名场面数：[X] 个
```

如有不合格章节（超过3次重写仍未通过），在报告中标注：
```
⚠️ 第X章：字数不足（[实际字数]/2000），已重试3次
```
